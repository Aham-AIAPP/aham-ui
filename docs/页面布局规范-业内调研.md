# 业内页面布局规范调研 — 完整维度清单 + Aham 草案缺口（v2 终稿）

> 本稿在「综合稿」基础上，吸收对抗式审稿的全部 missingDimensions / webAppConflicts / weakSpots / suggestedAdditions：补齐 i18n/RTL、可变字号缩放、软键盘、虚拟列表、首屏折叠线、网格 vs flex vs subgrid、z-index 层叠、嵌入/iframe、邮件 HTML、打印导出、hover/pointer、布局动效降级等遗漏维度；裁决并消解多处「应用规则误套网页」「断点三套并存无 source of truth」「裸 px 与建议 rem 自相矛盾」等冲突；并给所有 [自定] 取值补上栅格推导依据。面向 Aham UI 维护者，目标是可直接据此写下一版「网页轨 / 应用轨 / Office 轨 / 邮件轨 + 通用」的多轨布局规范。

---

## 一、结论速览

**直接回答「是不是改了半调子、考虑不全？」—— 是，确实考虑不全，而且是「系统性地只覆盖了一半的世界」。** 现有草案 `页面布局规范-审计与草案.md` 内层很扎实，但它几乎完全建立在 **Apple HIG（macOS 桌面应用）单一来源**之上：

1. **只有「应用轨」，没有「网页轨」。** Aham 自我定位是「同时落地 macOS 桌面应用 + 网页 + Office」，但草案的断点、容器、栅格、导航壳、响应式策略全部按「窗口/桌面 app」心智写。网页侧最核心的规范——**12 列栅格、阶梯式 container max-width、容器居中、mobile-first min-width 级联、容器查询**——草案要么没有，要么用 HIG 的「可读宽度」含糊带过。
2. **居中/最大宽度这条主轴自相矛盾、且选错默认。** 草案写「默认左对齐、不要居中」（抄自 HIG），但同一文档 examples 里**所有页面都是 `margin:0 auto` 居中**（网页惯例）。草案没意识到这是「网页轨 vs 应用轨」的根本分叉，把网页正确做法当成「方向相反的错误」去纠正。
3. **页面级骨架抽象缺一层。** 缺业内公认的两层中间抽象：**Pane 模型（1–3 窗格）** 与 **Canonical Layouts（List-Detail / Supporting-Pane / Feed）**——正是「供 AI 消费」最需要的「页面长什么样」成品蓝图。
4. **断点哲学停留在 size-class，没建「窗口宽度 → 结构」的可执行映射表，且本身三套断点并存（见第三节冲突修正）。**
5. **若干维度业内标配、草案完全没有：** 容器查询、内容密度档位、骨架屏白/黑名单、超宽屏处理、滚动吸顶偏移、阅读顺序↔DOM 源序约束、Office 落地布局（第三轨）。
6. **【本轮新增的元结论】** 即便补齐上述六点，仍有至少 12 个布局维度被遗漏或降格：**i18n/RTL（被当无障碍脚注）、可变字号/Dynamic Type 缩放、软键盘/IME、虚拟列表/滚动性能、首屏折叠线、网格 vs flex vs subgrid 取舍、z-index 层叠、嵌入/iframe、邮件 HTML（第四轨）、打印/PDF 导出、hover/pointer 能力、布局形变动效/reduced-motion**。不补这些，只是把「单轨不完整」升级成「双轨仍不完整」。

## 二、一份完整的页面布局规范应覆盖哪些维度（业内 taxonomy）

把全部来源的 `dimensionsCovered` 归并去重，并补入本轮新增维度，得到 **15 组、约 48 个维度**。每条一句话说明「要规定什么」（✚ = 本轮审稿补入）：

**A. 网格与间距基底**
1. **基础单位**：4px 或 8px 网格（Fluent 4px、Material/Carbon/Ant 8px、KDE gridUnit 18px、WinUI 4epx）。
2. **spacing 标度**：一套离散 token（2/4/8/12/16/24/32/48/64/80/96），区分「组件内 padding」与「区块间 layout spacing」。
3. **栅格列数**：网页 12 列（Bootstrap/Tailwind/Atlassian/Fluent/GOV.UK），Carbon 16 列，Ant 24 列；列数随断点跳变（4→8→12 或 4→8→16）。
4. **gutter / margin**：列间距与页边距，须落在间距刻度，且**随断点放大**（小屏 12-16，大屏 24-40），二者可不相等。
5. **媒体长宽比**：标准 aspect ratio 集（1:1/16:9/4:3…），行高=列宽倍数。
6. ✚ **布局机制取舍（Grid vs flex vs subgrid）**：何时用 CSS Grid（二维/页面骨架/卡片墙）、flexbox（一维/工具栏/按钮组）、subgrid（卡片内元素跨卡片基线对齐）——实现层决策，机读关键。

**B. 断点与响应式哲学**
7. **断点取值**：网页给 px/rem 媒体查询表，应用给 dp/pt 语义档；**须确定单一 source of truth，另一套派生（见第三节冲突修正）**。
8. **断点哲学**：按**窗口/容器宽度**而非设备类型；网页 mobile-first min-width 级联（含「sm: ≠ 手机」陷阱），桌面应用可 max-width 逐级折叠；HIG 用 regular/compact 二元。
9. **高度断点**：矮窗口/横屏时是否显顶栏、收纵向间距。
10. **响应式策略（reflow 动词）**：流式为主 + 断点处结构跳变 + max-width 封顶（hybrid）；Fluent/WinUI 五/六种 reflow 动词（重定位/缩放/重排/显隐/重构/替换）。
11. **容器查询（跨轨）**：可复用组件按**父容器宽度**响应——网页 @container/cqi，应用 SwiftUI containerRelativeFrame/ViewThatFits、Compose BoxWithConstraints（**统一为「组件按父容器响应」原则，非 Web only，见冲突修正**）。
12. ✚ **可变字号 / 缩放**：所有布局相对单位（rem/em/ch/cqi），容器随字号长高不溢出；网页 200% text zoom（WCAG 1.4.4）+ 400% reflow（1.4.10）不横向滚动；应用 Dynamic Type/sp 档位；触控目标/行高/控件高随字号联动。
13. ✚ **输入能力查询（pointer/hover）**：`pointer: coarse / any-hover` 下 hover-only 操作必须有常驻等价、命中区随输入方式放大（比密度档位更基础）。

**C. 容器与宽度**
14. **内容最大宽度 + 居中**：网页 container 阶梯 max-width + margin:auto；应用把多余宽度分配给新 pane（铺满）。
15. **正文阅读宽**：独立于页面宽，限 60–75 字符 / readableContentGuide（CJK 用 em/ch）。
16. **流式/固定/混合三态容器**：营销页定宽居中、控制台全宽/混合。
17. **超宽屏处理**：margin 收束居中 vs 「每次加 2 列」扩列。
18. ✚ **多显示器 / 多窗口分屏 / 超宽比护栏**：macOS Split View / iPad 多任务下每窗口独立按容器宽响应；21:9/32:9 ultra-wide 最大内容宽护栏。
19. ✚ **嵌入 / iframe / 受限容器**：嵌入场景无视口断点，只能靠容器查询；固定尺寸沙箱、嵌入式 widget 最小可用尺寸。

**D. 页面骨架与页型**
20. **页面区域顺序**：skip link → header → main → footer，DOM 序即阅读序。
21. **Pane 窗格模型**：页面=1–3 个 pane（fixed/flexible/floating/semi-permanent，至少一个 flexible）。
22. **Canonical Layouts**：List-Detail / Supporting-Pane / Feed 三大页型模板，每种给紧凑/中/展开三套配置与栏比。
23. **页型模板库**：首页/仪表盘/列表/表格/详情/表单（Ant 6 类）。
24. **主/辅分栏比例**：详情页 2/3+1/3，supporting 展开 70/30、中 50/50（**须与 Aham 固定栏宽体系打通，见弱点修正**）。
25. ✚ **首屏 / 折叠线预算**：关键 CTA 与价值信息首屏可见、首屏高度预算、LCP 元素布局优先级。
26. ✚ **虚拟列表 / 滚动容器**：长列表虚拟化的行高约束、滚动容器固定高+overflow、sticky 分组表头与虚拟滚动协同、骨架占位高=实际行高防 CLS、body scroll lock。

**E. 导航壳**
27. **导航壳选择**：目的地数量阈值 + 屏宽占用（**阈值区间需去重定来源，见弱点修正**）。
28. **导航壳随断点形变**：一套 IA 映射底部 bar↔rail↔drawer / sidebar↔tab，含切换阈值。
29. **导航层级与深度**：≤2 级，超 2 级配面包屑；导航组件容量上限。
30. **侧栏折叠/overlay**：窄屏无动画自动折叠、overlay 覆盖而非推走、快捷键。

**F. 页头与工具栏**
31. **页头组成与顺序**：面包屑→标题→（副标题/tag）→操作（右）→tabs/工具栏。
32. **面包屑**：截断算法（首项+末两项）、与 back link 互斥、宽度护栏。
33. **页面操作配额**：≤1 primary + ≤3 secondary（**「禁页底重复」为应用轨规则，网页/移动轨豁免，见冲突修正**）。
34. **工具栏/搜索/筛选**：三段式、搜索落位、命令溢出。

**G. 内容密度与输入**
35. **密度档位**：comfortable/standard/compact，= 输入方式（触控/指针）×宽度档；同页不混密度（**须给三档实际取值，见弱点修正**）。

**H. 状态布局**
36. **空状态**：图+标题+正文+CTA，**对齐须单一裁决（见弱点修正）**，必须含「下一步」。
37. **加载/骨架屏**：skeleton 白名单（容器/数据类）/黑名单（按钮/输入/toast/弹窗）；>1s 才用；占位=最终尺寸防跳动（CLS）。
38. **错误/通知**：inline 置内容区顶部或就近，错误三重指示（红框+图标+文案），toast vs inline（**移动端 toast 通常在底部，见冲突修正**）。

**I. 全球化与本地化**
39. ✚ **i18n / RTL 镜像**：文本扩张预留（德/俄比中文长 30–40%，CJK 更短，按钮/标签/导航不写死宽度）；RTL 全镜像（栅格/导航壳 leading↔trailing 互换、面包屑/返回箭头/进度方向、icon 是否镜像、用 logical properties `margin-inline` 而非 `margin-left`）；数字/日期/货币格式占位。

**J. 状态层级与浮层**
40. ✚ **z-index / 层叠顺序**：base/sticky/dropdown/overlay/modal/toast 分层 token + stacking context 规则 + 滚动锁定（body scroll lock）。

**K. 动效与过渡**
41. ✚ **布局形变动效**：pane 折叠 / drawer 滑入 / 列表重排的过渡 + `prefers-reduced-motion` 降级 + view transitions + 避免布局抖动。

**L. 交互/输入边界**
42. ✚ **软键盘 / IME 弹出**：移动 web 与触屏应用软键盘压缩视口（visualViewport / `interactive-widget=resizes-content`），吸底按钮/表单提交避遮挡、focus 时 scroll-into-view。

**M. 可访问与边界**
43. **可访问阅读顺序**：DOM 源序=视觉序、skip link、Tab 焦点、RTL、400% 缩放降级、aria-live。
44. **安全区/边界**：safe-area / env(safe-area-inset-*)（**网页前提 viewport-fit=cover，见冲突修正**）、避开 sticky 遮挡、viewport meta。
45. **最小点击目标 + 滚动/吸顶**：≥44/48；sticky 页头给内容偏移量。

**N. 跨档状态**
46. **跨档状态保持**：折叠/旋转/分屏保留选中态+滚动位+折叠 back（App 为主）。

**O. 输出态轨道（非屏幕）**

## 三、网页 vs 应用：关键差异 + 冲突修正（最重要一节）

这是草案的最大盲区。Aham 同时做两端（外加 Office、邮件），**必须分轨制定**。逐维度对比，并在表后逐条**裁决本轮审稿指出的「误套/含糊/绝对化/自相矛盾」**。

| 维度 | 网页轨（Web） | 应用轨（macOS/桌面 App） | 是否通用 |
|---|---|---|---|

| 断点心智 | mobile-first，min-width 级联（`sm:` = ≥640px，不是「手机」） | full-screen-first，逐级折叠（max-width / 晚折叠）；或 regular/compact 二元 | ❌ 分轨 |

| 吸底/固定页脚/sticky CTA | **合法且推荐**（吸底操作栏、移动底部主按钮、sticky footer） | **HIG 铁律：底部不放关键控件** | ❌ **冲突点，须分轨豁免** |

### 冲突逐条裁决（吸收 webAppConflicts）

**C1 — 居中/最大宽度冲突（明确裁决，务必写进草案）：**
- **网页轨**：「页面壳层」用阶梯 max-width + `margin:0 auto` 居中收口（网页惯例，examples 已这么做，**是对的，不要去「纠正」它**）。区分两个独立宽度：**页面最大宽**（壳，宽）与**正文阅读最大宽**（窄，60–75ch）。超宽屏用 margin 继续放大居中，或数据密集页 full-width / 加列。
- **应用轨**：内容**左对齐铺满**工作区，多余宽度展开第二/第三 pane；仅单任务全屏页（登录/向导/空态）才居中，正文用 readableContentGuide 收窄。

**C2 — 「底部不放控件」误套网页（修正为分轨规则）：** HIG 的 avoid-bottom-of-window 是 **macOS 应用规则**，原综合稿在 W6/通用照抄，错误。**裁决**：应用轨保留该铁律；**网页轨/移动轨显式豁免**——sticky CTA、吸底操作栏、固定页脚、移动端底部主按钮（拇指热区）均合法推荐。配套：吸底元素须给 safe-area 内边距 + 软键盘弹起让位规则。错误/通知 inline 默认置内容区顶部，但**移动端瞬态 toast 通常在底部**，分轨写明。

**C3 — 断点三套并存无 source of truth（修正）：** 原稿网页 640/768/1024/1280/1536（Tailwind px）、应用 600/840/1200/1600（Material dp）、草案 sm380/md860/lg1280 并存，机读会混乱。**裁决**：
- **以网页轨 rem 媒体查询为唯一 source of truth**：`sm 40rem(640) / md 48rem(768) / lg 64rem(1024) / xl 80rem(1280) / 2xl 96rem(1536)`。
- **应用轨 dp 语义档作派生映射**，明确换算口径：dp 视为「密度无关像素」，在 1× 下数值≈px，故 Material 600/840/1200/1600 对应网页 ~37.5rem/52.5rem/75rem/100rem，**取最近网页档对齐**（600≈sm-md 之间、840≈lg、1200≈xl、1600≈2xl 之上）。
- **废止草案原 sm380/md860/lg1280 三套并存**；其中 380 仅作「极窄压缩」特例标注、860 映射到应用轨主切换点（≈Medium/Expanded 边界），不再作为独立第三套。

**C4 — 容器查询错标 Web only（修正）：** 应改标**跨轨可用**，统一为「组件按父容器响应」原则：网页 `@container` + `cqi`；应用 SwiftUI `containerRelativeFrame`/`ViewThatFits`、Compose `BoxWithConstraints`。这正支撑「供 AI 拼装、同卡片放主区/侧栏都正确」——应用轨同样需要。

**C5 — 网页 token 用裸 px 自相矛盾（修正）：** 原稿主张「建议 rem / 正文 rem 允许缩放」，却把 W1 `page-max:1280`、W2 `gutter:24` 写成裸 px。**裁决**：网页轨容器宽/正文宽/gutter/margin **一律 rem（正文宽辅以 ch）**，px 仅保留给 1px 描边等非缩放量。具体取值见第六节（已补 rem 表达 + 栅格推导）。

**C6 — 网页轨内部需再分桌面/移动（修正）：** 原稿「网页轨默认顶栏、窄屏坍缩汉堡」把桌面 web 规则当成整个网页轨。**裁决**：网页轨再分 **桌面 web**（默认顶部水平导航）与 **移动 web**（底部 tab 为主流合法模式 + 顶部精简栏）；导航壳形变表须覆盖这两段。

**C7 — 安全区给规则没给前提（修正）：** 网页用 `env(safe-area-inset-*)` 的**前提是 `<meta viewport ... viewport-fit=cover>`**，否则 env 恒为 0。规则与触发条件须成对写出。

**C8 — Office 不是单一「固定画布」（修正）：** 见第五节缺口 #10 与第六节 Office 轨——拆为 Word / Excel / PPT 三个子模型，不再笼统称「固定画布」。

### Office 轨与邮件轨的分轨定位（占位）
- **Office 轨（第三轨，业内无现成参考，需 Aham 自定）**：**不可笼统称「固定画布」**，须拆 Word（流式文档，接近网页内容轨）/ Excel（单元格网格，自有坐标系）/ PPT（固定画布幻灯片）三子模型，各按其布局心智声明内容安全区 + 与品牌网格对齐。
- ✚ **邮件 HTML 轨（第四轨，若产品矩阵含通知/营销邮件）**：完全独立——table 布局、内联样式、不保证 flex/grid/媒体查询、600px 固定宽、暗色模式 meta；与网页轨明确区分，先占位声明。

## 四、各体系亮点速记

- **Material Design 3 / Android adaptive**：① 三大 **Canonical Layouts** + 各档栏比；② 「第二区内容能否独立成立」判定 list-detail vs supporting-pane；③ 一套导航项三种壳（bar→rail→drawer）映射 + 宽 margin 收束（最大 200dp）；④ 跨配置状态保持写进规范。
- **IBM Carbon**：① 「每次加 2 列」超宽屏可控扩列（>1584px）；② 三种 gutter 模式（wide/narrow/condensed）；③ 间距 token 跨断点固定但允许「跳档」调密度；④ skeleton 白/黑名单。
- **Ant Design**：① **数学化间距公式**（gutter=16+8n、Header=48+8n、Sider=200+8n）——最契合「供 AI 消费」；② 24 列细分栅格；③ 6 类页型模板。
- **Shopify Polaris**：① 内容宽度命名档位（small/base/large）+「仅 base 才有 aside」；② 页面操作硬上限（1 primary + ≤3 secondary）+ 禁页底操作（**注意此为应用式约束，网页豁免**）；③ 空状态定义为「整页级」。
- **Microsoft Fluent / WinUI**：① 五/六种 reflow 动词——把「做响应式」变成可执行清单；② 命令溢出做成一等机制（最小屏 320epx 仅容 4 primary）；③ epx 有效像素；④ 页头 52px 作滚动裁剪点。
- **GOV.UK + NN/G**：① **面包屑与 back link 绝不同页共存**；② 正文恒放 two-thirds 列锁行宽 ≤75 字符；③ mobile-first 反作用 content dispersion——桌面须主动再致密化；④ skip link 作页面首个可聚焦元素。
- **Atlassian**：① 栅格仅作用主内容区（排除 SideNav/Aside）；② gutter+margin 随断点同步放大（12/16→16/32）；③ 导航三基元收敛；④ 顶栏只放跨产品通用动作。
- **Apple HIG**：① sidebarAdaptable 一套架构两种外壳；② 变窄优先砍第三栏（inspector）；③ readableContentGuide ≠ 整页 max-width；④ 工具栏 trailing 项永不折叠。
- **GNOME/KDE**：① split header bars（每栏独立页头）；② sp 单位（断点随字号缩放）；③ AdwClamp 600px 内容宽上限；④ sidebar vs utility pane 语义区分。
- **Bootstrap/Tailwind/Bulma**：① 三态容器（fixed/fluid/hybrid）；② `is-max-desktop` 封顶防超宽行长；③ 容器查询；④ 移动端默认堆叠、分列是渐进增强。
- ✚ **WCAG / MDN（缩放与本地化）**：① 1.4.4 text-only 200% zoom 与 1.4.10 reflow 400% 不横向滚动是两条独立判据；② CSS logical properties（margin-inline/inset-inline）是 RTL 全镜像的工程手段；③ `prefers-reduced-motion`、`pointer:coarse`/`any-hover`、visualViewport、`@container`、view transitions 等媒体/能力查询是布局适配一等机制。

## 五、Aham 草案的缺口（直击「考虑不全」）

### 【完全没想到】的维度

1. **网页 vs 应用分轨（最大缺口）。** 现状：单轨、把网页居中当错误纠正。**Aham 应补**：每个页面模板显式声明属「网页轨（居中收口）/ 应用轨（铺满成栏）」；给 source-of-truth 断点表（网页 rem 主、应用 dp 派生，见 C3）；网页轨内部再分桌面/移动（见 C6）。

2. **栅格列数。** **Aham 应补**：网页轨 12 列栅格（gutter 锚定 spacing token，随断点 16/24/32）；应用轨「最小卡片宽自适应列」。所有取值须由「12 列 × 列宽 + 11 gutter + 2 margin」反推（见弱点修正 W1）。

3. **容器查询。** **Aham 应补**：明文「页面响应用视口/窗口断点，组件响应用容器查询」——**跨轨通用**（网页 @container/cqi，应用 containerRelativeFrame/ViewThatFits/BoxWithConstraints，见 C4），是机读系统最该领先的一条。

4. **Pane 模型 + Canonical Layouts。** **Aham 应补**：定义 `pane`（fixed/flexible/floating，至少一个 flexible）；落 3 个页型成品（list-detail / supporting-pane / feed），各给紧凑/中/展开三套配置与栏比；**栏比须与 Aham 既有固定栏宽（sidebar264/rightbar340）统一打通（见弱点修正 W4）**。

5. **内容密度档位。** **Aham 应补**：三档密度 token，且**给出每档实际取值**（行高/控件高/间距）；规则「触控窄屏 comfortable、指针宽屏可 compact，同页不混」。

6. **加载/骨架屏布局。** **Aham 应补**：`.skeleton` 规则 + 白黑名单 + 「占位尺寸=最终尺寸防 CLS」；与虚拟列表协同（见 ✚23）。

7. **超宽屏处理。** **Aham 应补**：网页轨默认 margin 收束居中；数据密集页 full-width 或 lg 之上扩列；✚ 补多显示器/多窗口分屏每窗口独立按容器响应、21:9/32:9 内容宽护栏。

8. **导航壳随断点形变映射表。** **Aham 应补**：一张「宽度档 → 导航组件」表，网页桌面默认顶栏、网页移动默认底栏、应用默认侧栏/底栏，给切换阈值（阈值来源须去重，见弱点修正 W2）。

9. **跨档状态保持。** **Aham 应补**：master-detail 折叠的状态迁移（保留选中+滚动，back 从详情回列表）+「大屏无选中显占位详情」。

10. **Office 落地布局（第三轨）。** **Aham 应补**：**拆 Word / Excel / PPT 三子模型**（流式文档 / 单元格网格 / 固定画布），分别声明安全区与品牌网格对齐——不可笼统称「固定画布」（Excel 根本不是固定画布）。

— 以下为本轮审稿补入的【完全没想到】维度 —

11. ✚ **i18n / RTL 布局。** 现状：草案与综合稿都只在阅读顺序里点一句 RTL。**Aham 应补**：跨轨小节——文本扩张预留（按钮/标签 min-width 不写死、`max-content` + 截断策略）；RTL 全镜像（用 CSS logical properties `margin-inline`/`padding-inline`/`inset-inline`，禁用 `left/right`；导航 leading↔trailing 互换、面包屑与返回箭头方向、进度方向、哪些 icon 镜像哪些不镜像）；CJK 行宽用 em/ch 而非字符数硬套；数字/日期/货币格式占位。

12. ✚ **可变字号 / 缩放。** 现状：仅一句「rem 允许缩放」。**Aham 应补**：跨轨小节——全部布局相对单位（rem/em/ch/cqi），容器随字号长高不溢出；网页同时满足 200% text zoom（WCAG 1.4.4）与 400% reflow（1.4.10，不横向滚动）；应用轨 Dynamic Type / sp 档位；触控目标/行高/控件高随字号联动。

13. ✚ **软键盘 / IME 弹出。** 现状：完全缺失（用户点名）。**Aham 应补**（网页移动轨 + 应用触控轨）：`visualViewport` / `interactive-widget=resizes-content`、聚焦输入时 `scroll-into-view`、吸底操作栏在键盘弹起时上移或让位。

14. ✚ **虚拟列表 / 滚动容器。** 现状：讲了 skeleton 没讲承载它的滚动容器。**Aham 应补**：长列表虚拟化行高约束（行等高/已知高）、滚动容器固定高+overflow、sticky 分组表头与虚拟滚动协同、骨架占位高=实际行高防 CLS、body scroll lock。

15. ✚ **首屏 / 折叠线预算。** 现状：有高度断点无折叠线。**Aham 应补**：关键 CTA 与价值信息首屏可见、首屏高度预算（与高度断点联动但不同）、LCP 元素布局优先级。

16. ✚ **网格 vs flex vs subgrid 取舍。** 现状：完全缺失。**Aham 应补**：实现层决策表——Grid 用于页面骨架/卡片墙/二维对齐，flex 用于工具栏/按钮组/一维流，subgrid 用于卡片内元素跨卡片基线对齐；给 AI 选择规则（机读关键）。

17. ✚ **z-index / 层叠。** 现状：js 提了 z-index 未成体系。**Aham 应补**：层叠 token 阶梯（base/sticky/dropdown/overlay/modal/toast）+ stacking context 规则 + 滚动锁定。

18. ✚ **嵌入 / iframe / 受限容器。** 现状：完全缺失（Aham 落地 Office + 组件供 AI 拼装必然出现）。**Aham 应补**：嵌入内无视口断点只靠容器查询、固定尺寸沙箱、嵌入式 widget 最小可用尺寸。

19. ✚ **邮件 HTML（第四轨）。** 现状：连占位都没有。**Aham 应补**：第四轨占位——table 布局/内联样式/600px 固定宽/暗色 meta，明确与网页轨不同。

20. ✚ **打印 / PDF 导出。** 现状：缺失（Aham 涉 Office/文档导出）。**Aham 应补**：`@media print`、分页 `break-inside`、隐藏导航/交互件、导出样式。

21. ✚ **hover / pointer 能力。** 现状：仅作密度因子。**Aham 应补**：`pointer:coarse`/`any-hover` 下 hover-only 操作必须有常驻等价、命中区随输入方式放大（比密度更基础）。

22. ✚ **布局形变动效 / reduced-motion。** 现状：讲了 reflow 动词没讲动效。**Aham 应补**：pane 折叠/drawer 滑入/列表重排的过渡 + `prefers-reduced-motion` 降级 + view transitions + 避免布局抖动。

### 【想到但太单薄】的维度

23. **断点。** **应补**：网页轨 rem 业界基线（640/768/1024/1280/1536，**作 source of truth**），应用轨 dp 语义档（600/840/1200/1600，**作派生映射**）+ 高度断点（<480 收顶栏）；明写「sm: ≠ 手机」陷阱；**废止 sm380/md860/lg1280 三套并存（见 C3）**。

24. **gutter/margin。** **应补**：gutter（列间）与 margin（页边）是两个 token、随断点同步放大、二者可不等。

25. **正文阅读宽 vs 页面宽。** **应补**：明文「页面壳最大宽（宽）与正文阅读宽（窄，60–75ch）独立设定」，给 GOV.UK two-thirds 列锁行宽做法；**正文宽用 ch 表达，不用裸 px（见 C5）**。

26. **空状态。** **应补**：结构（图标/插画 + 标题 + 正文 + 主 CTA）、**对齐单一裁决**（见弱点修正 WS-空状态：默认整页级居中，列表内嵌空态左对齐随容器）、必须含「下一步行动」。

27. **错误/通知（页面内）。** **应补**：inline 通知置内容区顶部/就近，错误三重指示（红框+图标+文案），toast（瞬态，**移动端常在底部**）vs inline（任务流内）区分。

28. **面包屑。** **应补**：截断算法（首项+末两项，中间进 overflow）、**与 back link 互斥不同页共存**（GOV.UK 硬规则）、宽度护栏。

29. **滚动/吸顶。** **应补**：sticky 页头/工具栏需给主体内容 padding 偏移防遮挡；fixed/sticky 导航预留偏移量。

30. **可访问阅读顺序。** **应补**：硬约束「响应式视觉重排不得破坏 DOM 源序与 Tab 焦点序」（慎用 flex order/row-reverse）、skip link 作首个可聚焦元素、viewport meta、400% 缩放降级、aria-live。

31. **安全区。** **应补**：网页 `env(safe-area-inset-*)`，**前提声明 `viewport-fit=cover`（见 C7）**。

32. **媒体长宽比。** **应补**：标准 ratio token 集（1:1/4:3/3:2/16:9/2:1）+「行高=列宽倍数遵循 ratio」（Carbon）。

## 六、给 Aham 的下一版草案补全建议

> 标注：**[自定]** = Aham 自定取值；**[↔业内]** = 结构/规则源自业内并注明出处；**[推导]** = 取值由栅格公式反推，非凭空。所有 **[自定]** 网页取值已尽量改为 [推导] 或标明派生依据，以纠正「取值无依据」的自指控（见弱点修正）。

### 网页轨（Web track）

> 网页轨内部再分 **桌面 web** 与 **移动 web**（见 C6）。所有容器/正文/gutter/margin 用 **rem**（正文辅以 ch），px 仅限非缩放量（见 C5）。

**W1 容器与居中（修正现有矛盾 + 取值推导）** [↔Bootstrap/Carbon/GNOME] [推导]
- 页面壳层：阶梯 max-width + `margin:0 auto` 居中收口（**网页正确做法，保留 examples 现状**）。
- **取值推导**（消除「凭空 1280」之嫌）：以 12 列 + `gutter 24px` + `margin 32px` 为例，列宽 `c`：`page-max = 12c + 11×24 + 2×32 = 12c + 328`。取列宽 c≈79px → `page-max ≈ 1276 ≈ 80rem(1280)`；内容区（扣页边）`page-max-content ≈ 12c + 11×24 = 1212 ≈ 75.75rem`，取 `--page-max-content: 67.5rem(1080)` 对应 lg-xl 间收口。**所有壳档随实际列宽/gutter 选定后回填，禁止凭空填数。**
- **[推导]** `--page-max: 80rem(1280)` / `--page-max-content: 67.5rem(1080)`。
- 正文阅读宽**独立**：`--read-max: 65ch`（≈ 45rem/720px，按字体回填），与页面宽分开。
- 超宽屏：默认 margin 收束居中；数据密集页 `--page-full`（full-width）；✚ 多显示器/分屏每窗口按容器宽独立响应；✚ 21:9/32:9 设 `--page-max-ultra` 护栏。

**W2 栅格 + 布局机制** [↔业界 12 列 / Atlassian] [推导]
- 12 列流动栅格；**gutter 与 margin 分离、随断点放大**：`--gutter: 1.5rem(24)`(lg) / `1rem(16)`(md) / `0.75rem(12)`(sm)；`--margin: 2rem(32)`(lg) / `1.5rem(24)`(md) / `1rem(16)`(sm)。
- 列数随断点：sm 单列堆叠（渐进增强分列）、md 8 概念列、lg 12 列。
- 栅格仅作用主内容区，不含 rail/sidebar/rightbar。
- ✚ **机制取舍**：页面骨架/卡片墙/二维对齐用 **CSS Grid**；工具栏/按钮组/一维流用 **flex**；卡片内元素跨卡片基线对齐用 **subgrid**。给 AI 一张选择决策表。

**W3 断点（rem，mobile-first，唯一 source of truth）** [↔Tailwind/USWDS]
- `sm:40rem(640) / md:48rem(768) / lg:64rem(1024) / xl:80rem(1280) / 2xl:96rem(1536)`。
- ✚ **高度断点**：视口高 `<30rem(480)` 收顶栏/收纵向间距。
- 明文陷阱：`sm:` = 「≥640px」不是「手机」；移动样式写无前缀基线。
- **应用轨 dp 档由本表派生（见 A2、C3）；草案旧 sm380/md860/lg1280 废止**。

**W4 容器三态** [↔Bootstrap/Bulma]
- `fixed`（定宽居中，营销/内容页）/ `fluid`（全宽，控制台/数据）/ `hybrid`（断点前全宽、之后定宽）。页型绑定容器类型。
- ✚ **嵌入态**：iframe/embed 内无视口断点，强制走容器查询；声明 widget 最小可用尺寸。

**W5 容器查询（跨轨原则的网页实现）** [↔CSS @container]
- 可复用组件用容器查询（`container-type:inline-size`，子用 `@container` + `cqi`）；页面用视口断点。**机读系统优先此条**（应用轨等价能力见 A3 末）。

**W6 导航壳（桌面/移动分写 + 吸底豁免）** [↔Atlassian/Fluent/NN/G]
- **桌面 web**：默认顶部水平导航；窄屏坍缩汉堡/全屏 drawer。sticky 顶栏给主体 `padding-top` 偏移。
- ✚ **移动 web**：底部 tab 为合法主流（拇指热区）+ 顶部精简栏。
- 目的地数量阈值见弱点修正 W2（去重后单一来源）。

**W7 状态布局（含移动端豁免）** [↔Carbon/Polaris/Atlassian]
- 空状态：默认**整页级居中**（单一裁决，见弱点修正）；列表内嵌空态随容器左对齐。图+标题+正文+主 CTA，必须含下一步。
- 骨架屏：白名单（卡片/列表/表/数据）、黑名单（按钮/输入/toast/弹窗/下拉），>1s 才用，占位=最终尺寸；与虚拟列表协同（W11）。
- 错误/通知：inline 置内容顶部/就近，三重指示；toast（移动端常在底部）vs inline 区分。

**W8 可访问 + 缩放 + i18n（强化）** [↔GOV.UK/MDN/WCAG]
- skip link 作首个可聚焦元素；DOM 源序=视觉序硬约束（慎用 flex order/row-reverse）；viewport meta（含 `viewport-fit=cover` 作 safe-area 前提）；同时满足 200% text zoom 与 400% reflow（不横向滚动）；aria-live 宣告加载；`env(safe-area-inset-*)`。
- ✚ **i18n/RTL**：用 logical properties（`margin-inline`/`inset-inline`），禁 `left/right`；按钮/标签不写死宽度（`max-content`+截断）；导航/面包屑/箭头/进度 leading↔trailing 镜像；CJK 行宽用 ch/em。

**W9 ✚ 软键盘 / 吸底（移动 web）** [↔MDN visualViewport]
- 用 `visualViewport` 或 `<meta name="viewport" content="...interactive-widget=resizes-content">`；聚焦输入时 `scroll-into-view`；吸底 CTA/操作栏在键盘弹起时上移或让位，并叠加 safe-area 内边距。

**W10 ✚ z-index / 层叠** [↔通用]
- 层叠 token 阶梯：`--z-base:0 / --z-sticky:100 / --z-dropdown:200 / --z-overlay:300 / --z-modal:400 / --z-toast:500`；每个浮层族建立独立 stacking context；modal/drawer 打开时 body scroll lock。

**W11 ✚ 虚拟列表 / 滚动容器** [↔通用]
- 长列表虚拟化：行等高或已知高、滚动容器固定高+overflow；sticky 分组表头与虚拟滚动协同（表头脱离虚拟窗口或单独层）；骨架占位高=实际行高防 CLS。

**W12 ✚ 布局动效 / 打印** [↔WCAG/print]
- 形变（pane 折叠/drawer/列表重排）走过渡或 view transitions，`prefers-reduced-motion` 降级为瞬时；
- `@media print`：隐藏导航/交互件、`break-inside:avoid` 控分页、容器转 100% 宽。
